# Phase 10: Announcements, Push & Alerts - Research

**Researched:** 2026-04-04
**Domain:** Firebase Cloud Messaging (FCM) + Cloud Functions v2 + React Web Push
**Confidence:** MEDIUM

## Summary

Phase 10 implements admin-authored announcements with municipality-scoped push delivery and citizen-facing alerts. The core challenge is bridging Firebase Cloud Messaging's topic-based pub/sub model with the project's municipality-scoped targeting requirement. Three architectural decisions drive everything: (1) FCM tokens are stored per-user in a subcollection so delivery can be tracked per-recipient, (2) users are subscribed to municipality topics on login based on their `municipalityCode` custom claim, and (3) announcement publishing triggers a CF that fans out to all targeted users' tokens via `sendEach`.

The existing `Announcement` type needs one extension: `targetScope.municipalityCode` must be an array (not a single string) to support multi-municipality announcements. The `AnnouncementStatus` enum (draft/published/cancelled) maps cleanly to a CF state machine. Client-side FCM requires a standalone `firebase-messaging-sw.js` in `public/` (not generated by vite-plugin-pwa's Workbox) plus foreground message handling in the app.

### Pattern 1: FCM Token-per-User with Delivery Tracking

**What:** Every authenticated user's FCM registration token is stored in a subcollection: `users/{userId}/fcmTokens/{tokenId}` with fields `token`, `createdAt`, `enabled`, `municipalityCode`.

**Why not topics for delivery:** FCM topic sends do not return per-recipient delivery status. Topics are fire-and-forget fanout. To satisfy ALR-06 (per-recipient delivery tracking), tokens must be stored and sent to individually.

**Why not just topic sends:** Topic sends are optimal for high-throughput broadcast (weather alerts). But the project needs delivery receipts. The hybrid approach: users subscribe to topics for receiving, but the CF sends to each user's stored tokens individually for tracking.

**Flow:**
1. User logs in → `useFcmToken` hook calls `getToken(messaging, {vapidKey})`
2. Token saved to `users/{userId}/fcmTokens/{autoId}` with `municipalityCode`
3. CF `sendAnnouncementPush` queries all `users` with matching `municipalityCode`, reads their tokens, calls `messaging.sendEach(messages)`
4. For each token's result, upserts `announcements/{announcementId}/notifications/{userId}` with `status: 'delivered'|'failed'`

**Example:**
```typescript
// Client: src/lib/firebase/messaging.ts
import { getToken } from 'firebase/messaging'
import { messaging } from '@/lib/firebase/config'

export async function saveFcmToken(userId: string, municipalityCode: string): Promise<string | null> {
  const token = await getToken(messaging, { vapidKey: import.meta.env.VITE_FCM_VAPID_KEY })
  if (!token) return null

  await addDoc(collection(db, 'users', userId, 'fcmTokens'), {
    token,
    municipalityCode,
    enabled: true,
    createdAt: serverTimestamp(),
  })
  return token
}

// CF: functions/src/announcements/sendAnnouncementPush.ts
import { getMessaging } from 'firebase-admin/messaging'
import { getFirestore } from 'firebase-admin/firestore'

export async function sendAnnouncementPush(announcementId: string, targetScope: AnnouncementTargetScope) {
  const db = getFirestore()
  const messaging = getMessaging()

  // Collect all user tokens for the target scope
  const tokens: string[] = []
  const userIds: string[] = []

  let userQuery: FirebaseFirestore.Query
  if (targetScope.type === 'province') {
    userQuery = db.collection('users')
  } else {
    userQuery = db.collection('users').where('municipalityCode', 'in', targetScope.municipalityCode)
  }

  const users = await userQuery.get()
  for (const userDoc of users.docs) {
    const tokensSnap = await db.collection('users', userDoc.id, 'fcmTokens')
      .where('enabled', '==', true).get()
    for (const tokenDoc of tokensSnap.docs) {
      tokens.push(tokenDoc.data().token)
      userIds.push(userDoc.id)
    }
  }

  if (tokens.length === 0) return { sent: 0 }

  const message: MulticastMessage = {
    tokens,
    notification: { title: announcement.title, body: announcement.body },
    webpush: {
      notification: { icon: '/icons/icon-192.png', click_action: '/app/alerts' },
      fcmOptions: { link: '/app/alerts' },
    },
    data: { announcementId, type: announcement.type, severity: announcement.severity },
  }

  const response = await messaging.sendEachForMulticast(message)

  // Track per-recipient delivery
  const batch = db.batch()
  response.responses.forEach((res, i) => {
    const notifRef = db.doc(`announcements/${announcementId}/notifications/${userIds[i]}`)
    batch.set(notifRef, {
      status: res.success ? 'delivered' : 'failed',
      sentAt: FieldValue.serverTimestamp(),
      error: res.error?.message ?? null,
    }, { merge: true })
  })
  await batch.commit()

  return { sent: tokens.length, successCount: response.successCount, failureCount: response.failureCount }
}
```

### Pattern 2: Municipality Topic Subscription on Login

**What:** When a user authenticates, they are subscribed to FCM topics `municipality_{code}` (where `code` is their `municipalityCode`) and `province_CMN`. This enables topic-based receives even though delivery is tracked via tokens.

**Why:** FCM topic messaging is the standard way to route push messages to user segments without storing tokens server-side. Subscribing on login (not just registration) ensures topic membership stays current with the user's municipalityCode.

**Example:**
```typescript
// In AuthProvider.tsx or a useEffect after login
import { subscribeToTopic } from 'firebase/messaging'

async function subscribeUserToMunicipalityTopics(messaging: Messaging, municipalityCode: string) {
  if (!municipalityCode) {
    // Province-wide only
    await subscribeToTopic(messaging, 'province_CMN')
  } else {
    await subscribeToTopic(messaging, `municipality_${municipalityCode}`)
    await subscribeToTopic(messaging, 'province_CMN')
  }
}
```

### Pattern 3: Announcement State Machine

**What:** Announcements follow `draft → published → cancelled`. Only `published` triggers FCM delivery. Published announcements are immutable (no edits). Cancelled announcements remain visible but marked cancelled.

**Transition rules:**
- `draft → published`: Admin clicks "Publish" → CF validates, sets `publishedAt`, triggers `sendAnnouncementPush`
- `published → cancelled`: Admin clicks "Cancel" → CF sets `cancelledAt`, NO FCM retraction (FCM doesn't support message deletion)
- `draft → cancelled`: Admin clicks "Discard" → CF deletes doc or sets `cancelledAt`

```typescript
// Source: firebase-admin API (verified via messaging.d.ts)
export enum AnnouncementStatus {
  Draft = 'draft',
  Published = 'published',
  Cancelled = 'cancelled',
}
```

### Pattern 4: Standalone FCM Service Worker

**What:** A `public/firebase-messaging-sw.js` file (separate from VitePWA's Workbox service worker) handles background push messages.

**Why standalone:** VitePWA generates its own Workbox service worker. FCM requires its own standalone SW file registered at the domain root. Both can coexist: VitePWA's SW handles caching, FCM's SW handles push notifications.

**Critical:** The FCM SW file must be in `public/` (not `src/`) and named exactly `firebase-messaging-sw.js`. It is NOT processed by Vite.

**Source:** [Firebase Web Setup](https://firebase.google.com/docs/cloud-messaging/js/client)

```javascript
// public/firebase-messaging-sw.js (standalone, NOT compiled)
// Must use firebase/app compat import for SW context
import { initializeApp } from 'firebase/app'
import { getMessaging } from 'firebase/messaging/sw'
import { getPublicVapidKey } from 'firebase/messaging'

const firebaseApp = initializeApp({
  apiKey: "...",
  authDomain: "...",
  // ... same config as main app
})

const messaging = getMessaging(firebaseApp)

// Background message handler
self.addEventListener('push', (event) => {
  const data = event.data.json()
  const options = {
    body: data.notification.body,
    icon: '/icons/icon-192.png',
    badge: '/icons/icon-192.png',
    data: data.data,
    actions: [
      { action: 'view', title: 'View Alert' },
      { action: 'dismiss', title: 'Dismiss' },
    ],
  }
  event.waitUntil(self.registration.showNotification(data.notification.title, options))
})

self.addEventListener('notificationclick', (event) => {
  event.notification.close()
  if (event.action === 'view') {
    event.waitUntil(clients.openWindow('/app/alerts')))
  }
})
```

### Pattern 6: Firestore Security Rules for Announcements

**What:** Firestore rules for `announcements` collection (already in place at lines 96-103).

**Current rules:**
- Read: All authenticated users can read announcements
- Create: Superadmin OR municipal admin for the target municipality
- Update: Superadmin OR municipal admin for the target municipality
- Delete: Superadmin only

**Gap identified:** Rules allow create/update based on `targetScope.municipalityCode` in the resource, but ALR-01 requires ability to target multiple municipalities. The `AnnouncementSchema` uses `z.discriminatedUnion('type', ...)` with single `municipalityCode: z.string()`, which does not support arrays. This must be extended.

### Pattern 7: Announcement Scope Filter Query

**What:** Client queries `announcements` collection for published alerts matching user's scope.

**Query:**
```typescript
// Get alerts for a specific user's municipality
const announcementsQuery = query(
  collection(db, 'announcements'),
  where('status', '==', 'published'),
  where('targetScope.type', '==', 'province'),
  orderBy('publishedAt', 'desc'),
  limit(20)
)

// Also fetch municipality-scoped
const muniQuery = query(
  collection(db, 'announcements'),
  where('status', '==', 'published'),
  where('targetScope.type', '==', 'municipality'),
  where('targetScope.municipalityCode', 'array-contains', 'bas'),
  orderBy('publishedAt', 'desc'),
  limit(20)
)
```

**Note:** `array-contains` only matches arrays where the value appears. For multi-municipality announcements, `targetScope.municipalityCode` should be stored as an array. If single municipality, store as single-element array `['bas']`.

### Pattern 8: Announcement Multi-Municipality Extension

**What:** Extend `Announcement.targetScope` to support arrays for multi-municipality targeting.

**Current (single municipality only):**
```typescript
targetScope: {
  type: 'municipality' | 'province'
  municipalityCode?: string  // single string
}
```

**Required extension:**
```typescript
targetScope: {
  type: 'municipality' | 'province' | 'multi_municipality'
  municipalityCodes?: string[]  // array for multi-municipality
}
```

This enables ALR-02: provincial_superadmin can target any scope including multi-municipality, while municipal_admin is restricted to their own single municipality.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| FCM token management | Custom token storage API | Firestore subcollection `users/{uid}/fcmTokens` | Standard pattern; enables delivery tracking |
| Push delivery | FCM topic send (no tracking) | `messaging.sendEachForMulticast` to stored tokens | Per-recipient delivery status required by ALR-06 |
| Background push handling | VitePWA Workbox SW | Standalone `firebase-messaging-sw.js` in `public/` | FCM SW must be uncompiled at root; Workbox SW handles caching |
| Client FCM setup | Manually integrating messaging SDK | `firebase/messaging` modular SDK + `getToken(messaging, {vapidKey})` | Official SDK handles token refresh and SW registration |

**Key insight:** FCM delivery tracking is only possible when sending to individually stored tokens. Topic sends return only a fanout ID, not per-device results. The project MUST store tokens server-side to satisfy ALR-06.

## Common Pitfalls

### Pitfall 1: FCM VAPID Key Not Configured
**What goes wrong:** `getToken()` returns null or throws; push permission granted but no token generated.
**Why it happens:** VAPID key pair not generated in Firebase Console > Project Settings > Cloud Messaging > Web configuration.
**How to avoid:** Ensure `VITE_FCM_VAPID_KEY` env var is set in `.env.local`. The VAPID public key must be passed to `getToken()`.
**Warning signs:** Browser console: `FCM returned no token when subscribing the user to push`

### Pitfall 2: firebase-messaging-sw.js Not at Domain Root
**What goes wrong:** `getToken()` silently succeeds but push never works; background messages not handled.
**Why it happens:** VitePWA serves from `dist/`, but FCM SW must be at `/firebase-messaging-sw.js` (served from `public/`).
**How to avoid:** Place `firebase-messaging-sw.js` in `public/` directory. Vite copies `public/` files to dist root. Do NOT process it through Vite/TypeScript.
**Warning signs:** `Messaging: A problem occurred while subscribing the user to FCM: registration-error (messaging/token-subscribe-failed)`

### Pitfall 3: Topic Subscription Not Refreshed on Municipality Change
**What goes wrong:** User changes municipality (handled externally), but still receives alerts for old municipality.
**Why it happens:** Topic subscriptions persist in FCM until explicitly unsubscribed. User's stored `municipalityCode` changes but FCM topic membership doesn't.
**How to avoid:** On user login/profile load, read `municipalityCode` from custom claims, compare with stored subscription, unsubscribe from old topic and subscribe to new one.
**Warning signs:** Users receiving alerts for wrong municipality

### Pitfall 4: Array Query with `array-contains` on Non-Array Field
**What goes wrong:** Query with `where('targetScope.municipalityCode', 'array-contains', 'bas')` fails if `municipalityCode` is stored as a string (not array).
**Why it happens:** Firestore `array-contains` requires the field to be an array type.
**How to avoid:** Always store `municipalityCode` as an array in `targetScope` (single-element array for single-municipality: `['bas']`). Add a composite index for `announcements` by `status` + `targetScope.type` + `targetScope.municipalityCodes`.
**Warning signs:** Firestore error: `Invalid query. You cannot use 'array-contains' on field targetScope.municipalityCode because it is not an array field`

### Pitfall 5: FCM sendEach Limit (500 per batch)
**What goes wrong:** Sending to 1000+ users fails silently or times out.
**Why it happens:** `sendEach` accepts max 500 messages per call; fanout to large user base requires batching.
**How to avoid:** Chunk tokens into batches of 500 and call `sendEach` for each chunk with 1-second delay between chunks.
**Warning signs:** `messaging/invalid-argument` error when tokens array > 500

### Pitfall 6: Stale Tokens After App Reinstall
**What goes wrong:** Old token stored in Firestore still used for sends; push goes to old device.
**Why it happens:** FCM tokens can become invalid when the browser storage is cleared but the server-side token record persists.
**How to avoid:** Handle `messaging/token-refresh` event in the FCM SW and call `getToken()` to refresh. Also handle `onMessage` to detect undelivered messages and trigger token refresh.
**Warning signs:** High `failureCount` in delivery tracking; users reporting they don't receive alerts

### Cloud Function: Publish Announcement and Trigger Push
```typescript
// functions/src/announcements/publishAnnouncement.ts
// Source: firebase-admin v13.7.0 messaging API (verified via node_modules)
import { onCall } from 'firebase-functions/v2/https'
import { getFirestore, FieldValue } from 'firebase-admin/firestore'
import { getMessaging, MulticastMessage } from 'firebase-admin/messaging'

const PublishSchema = z.object({
  announcementId: z.string(),
})

export const publishAnnouncement = onCall(async (request) => {
  validateAuthenticated(request)
  const { announcementId } = PublishSchema.parse(request.data)

  const db = getFirestore()
  const announcementRef = db.collection('announcements').doc(announcementId)
  const announcementDoc = await announcementRef.get()

  if (!announcementDoc.exists) {
    throw new functions.https.HttpsError('not-found', 'Announcement not found')
  }

  const announcement = announcementDoc.data()!

  // Authorization: same rules as Firestore write rules
  const claims = request.auth!.token
  if (claims.role !== 'provincial_superadmin') {
    if (announcement.targetScope.type === 'municipality') {
      if (claims.municipalityCode !== announcement.targetScope.municipalityCode) {
        throw new functions.https.HttpsError('permission-denied', 'Cannot publish for this municipality')
      }
    }
  }

  // Update status to published
  await announcementRef.update({
    status: 'published',
    publishedAt: new Date().toISOString(),
  })

  // Trigger async push delivery (fire-and-forget from caller's perspective)
  // Full result tracked in notifications subcollection
  await sendAnnouncementPush(announcementId, announcement.targetScope)

  return { success: true, announcementId }
})
```

### Cloud Function: Send Push to All Targeted Users
```typescript
// functions/src/announcements/sendAnnouncementPush.ts
// Source: firebase-admin messaging API (verified)
import { getMessaging, MulticastMessage } from 'firebase-admin/messaging'
import { getFirestore, FieldValue } from 'firebase-admin/firestore'

const BATCH_SIZE = 500 // FCM sendEach limit

export async function sendAnnouncementPush(
  announcementId: string,
  targetScope: AnnouncementTargetScope
): Promise<{ sent: number; successCount: number; failureCount: number }> {
  const db = getFirestore()
  const messaging = getMessaging()

  // Get announcement data for push content
  const announcementDoc = await db.collection('announcements').doc(announcementId).get()
  const announcement = announcementDoc.data()!

  // Build user query based on target scope
  let userQuery: FirebaseFirestore.Query = db.collection('users')
  if (targetScope.type === 'municipality' && targetScope.municipalityCodes) {
    userQuery = userQuery.where('municipalityCode', 'in', targetScope.municipalityCodes)
  }

  const usersSnap = await userQuery.get()

  // Collect tokens
  const tokenBatches: string[][] = []
  const userIdBatches: string[][] = []

  let currentTokenBatch: string[] = []
  let currentUserIdBatch: string[] = []

  for (const userDoc of usersSnap.docs) {
    const tokensSnap = await db
      .collection('users').doc(userDoc.id)
      .collection('fcmTokens')
      .where('enabled', '==', true)
      .get()

    for (const tokenDoc of tokensSnap.docs) {
      const token = tokenDoc.data().token
      if (currentTokenBatch.length >= BATCH_SIZE) {
        tokenBatches.push(currentTokenBatch)
        userIdBatches.push(currentUserIdBatch)
        currentTokenBatch = []
        currentUserIdBatch = []
      }
      currentTokenBatch.push(token)
      currentUserIdBatch.push(userDoc.id)
    }
  }

  if (currentTokenBatch.length > 0) {
    tokenBatches.push(currentTokenBatch)
    userIdBatches.push(currentUserIdBatch)
  }

  // Send in batches
  let totalSuccess = 0
  let totalFailure = 0

  for (let i = 0; i < tokenBatches.length; i++) {
    const tokens = tokenBatches[i]
    const userIds = userIdBatches[i]

    const message: MulticastMessage = {
      tokens,
      notification: {
        title: announcement.title,
        body: announcement.body,
      },
      webpush: {
        notification: {
          icon: '/icons/icon-192.png',
          badge: '/icons/icon-192.png',
          click_action: '/app/alerts',
        },
        fcmOptions: {
          link: '/app/alerts',
        },
      },
      data: {
        announcementId,
        type: announcement.type,
        severity: announcement.severity,
      },
    }

    const response = await messaging.sendEachForMulticast(message)

    // Update delivery tracking
    const batch = db.batch()
    response.responses.forEach((res, idx) => {
      const notifRef = db.doc(`announcements/${announcementId}/notifications/${userIds[idx]}`)
      batch.set(notifRef, {
        status: res.success ? 'delivered' : 'failed',
        sentAt: FieldValue.serverTimestamp(),
        error: res.error?.message ?? null,
      }, { merge: true })
    })
    await batch.commit()

    totalSuccess += response.successCount
    totalFailure += response.failureCount
  }

  return { sent: tokenBatches.reduce((a, b) => a + b.length, 0), successCount: totalSuccess, failureCount: totalFailure }
}
```

## Open Questions

1. **Topic subscription on every login vs. only on registration**
   - What we know: Topic subscriptions persist in FCM. Re-subscribing on every login is safe (idempotent) but adds latency.
   - What's unclear: Whether there's a rate limit on topic subscription calls that could cause issues with frequent login/logout.
   - Recommendation: Subscribe on first login (when `users/{uid}/fcmTokens` is created) and on municipalityCode change (detected via custom claim comparison).

2. **Multi-municipality announcement UI**
   - What we know: ALR-01 requires multi-municipality targeting for provincial_superadmin.
   - What's unclear: Whether the CreateAlertForm should show a multi-select for municipalities or just "this municipality only" for municipal_admin and "select multiple" for superadmin.
   - Recommendation: Municipal_admin sees single municipality (pre-filled, disabled). Provincial_superadmin sees a multi-select checklist of all 12 municipalities.

3. **FCM delivery receipts vs. read receipts**
   - What we know: ALR-06 requires `pending/delivered/failed/read` per recipient. `delivered` means FCM confirmed delivery. `read` requires client-side click tracking.
   - What's unclear: Whether `read` status should be set when user opens the alert in the app (via TanStack Query `onSuccess`) or only when they click the notification.
   - Recommendation: `read` set when the announcement is fetched by a foreground listener or when the user navigates to the Alerts tab.

4. **VITE_FCM_VAPID_KEY environment variable**
   - What we know: VAPID key is required for web push. It must be generated in Firebase Console.
   - What's unclear: Whether the VAPID key is already generated and configured in the Firebase project for this app.
   - Recommendation: Verify before Phase 10 implementation. If not set, generate via Firebase Console > Project Settings > Cloud Messaging > Web Push Certificates.
